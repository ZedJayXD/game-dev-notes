# Obsidian AI 操作经验

> **适用对象**：WorkBuddy AI 助手
> **目的**：记录操作 Obsidian 笔记时经过实践验证的最佳操作方式
> **使用说明**：
> - 如需 AI 自行积累经验：仅复制下方「如何制定操作经验规范」部分发给 AI 即可
> - 如需 AI 了解现有经验：将下方全部内容（含具体经验）一起复制发给 AI

## 如何制定操作经验规范

> ⚠️ 以下为永久记忆级规范，AI 必须主动遵守，不得因上下文切换而遗忘。

操作经验是从实际使用中提炼出的"最佳实践"。制定时遵循以下规范：

1. **提炼标准**：一个操作被实践证明确实更优（更快/更稳/更少出错）时，才记录为操作经验
2. **优先级排序**：按"首选 → 次选 → 末选"排列，让执行者一眼知道该怎么做
3. **可复制性**：每条经验独立、自包含，单独拿出来也能执行
4. **禁止事项**：不仅要写"怎么做"，更要写"别怎么做"

## 已总结的操作经验（5 条）

> 📋 **以下为已总结完成的操作经验，需长期记忆**
> 
> - 如需自行积累经验：仅复制上方「如何制定操作经验规范」即可
> - 如需现成经验：可连带下方 5 条具体经验一起复制

### 🔧 经验 1：API 一直失败怎么办

当 Obsidian REST API 连续报错（401/404/乱码）时：

1. **首先检查连接**：用最简请求测试（`GET /vault/`），确认 API Key 格式是否正确（`Authorization: Bearer <key>`）
2. **换方法**：API 不行就直接读写文件系统（Obsidian 仓库路径下直接操作）
3. **不要死磕**：同一方法试 2 次还失败，立刻换方案，别循环同样的操作

**别这样做**：API 失败后反复重试同一个请求，每次只改参数顺序或格式，陷入循环。

### 🔧 经验 2：重命名文件的正确姿势

优先级顺序：

1. **首选**：直接文件系统重命名 → `os.rename(old, new)` 或 PowerShell `Rename-Item`
2. **次选**：Read 工具读内容 → Write 工具写新文件 → 验证内容正确 → 删除旧文件
3. **末选**：Obsidian REST API（只有在文件被 Obsidian 锁定、文件系统改不了时才用）

**别这样做**：先删旧文件 → 再建新文件（旧文件没了，新文件出问题就完了）

### 🔧 经验 3：验证新文件内容的标准流程

创建/修改完文件后，必须做以下验证：

1. 用 Read 工具重新读取刚写的文件
2. 检查前 5 行格式是否正确（frontmatter、`##` 标题、列表格式）
3. 检查是否有 `\n` 显示为文字的异常（格式崩溃的典型症状）
4. **确认无误后再动旧文件**

**别这样做**：写完文件不验证，直接宣告"完成"。

### 🔧 经验 4：Write/Edit 工具使用规范

- **Write 工具参数名**：`file_path` 和 `content`，不要写错成 `path`
- **Edit 工具**：必须先用 Read 工具读取文件，否则会报错
- **每次只改一个文件**：改完验证后再改下一个
- **编码问题**：Obsidian 文件永远是 UTF-8，写回时不要用 PowerShell `Out-File`/`Set-Content`

**别这样做**：Edit 工具不读文件直接用，或 Write 工具参数名写错导致报错。

### 🔧 经验 5：出错后的止损流程

一旦操作出错，按以下顺序处理：

1. **立即停止**当前操作，不要继续往下执行
2. **告诉用户**：发生了什么、可能的影响范围
3. **提出修复方案**，让用户确认后再执行，不要自作主张继续操作
4. **涉及删文件的操作**：必须先确认有无备份，才能执行

**别这样做**：出错后默默继续操作，或自行"修复"却造成二次损坏。
