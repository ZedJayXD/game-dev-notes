## 如何总结经验教训

> ⚠️ 以下为永久记忆级规范，AI 必须主动遵守，不得因上下文切换而遗忘。

经验教训不是"记住了就行"，而是要建立一套自动积累的机制。每次遇到问题或犯错时，按以下步骤总结：

1. **触发条件**：同一方法失败 2 次以上、排查时间过长、或导致数据损坏时，必须记录教训
2. **总结格式**：按「问题现象 → 错误原因 → 正确做法 → 预防措施」四步记录
3. **分类归档**：按类型（API 陷阱、文件操作、工具使用等）分类
4. **持续更新**：同类问题再次出现时，用新教训补充或修正旧教训，不重复记录
5. **主动应用**：每次开始操作前，先扫描相关教训，避免踩同样的坑

目标：让每次犯错都变成一次"永久免疫"，而不是"下次再犯"。

### ⚠️ 教训 1：REST API PUT 写文件时的格式陷阱

**问题现象**：通过 Obsidian REST API 的 PUT 端点创建文件时，文件里存了完整的 JSON 字符串，Obsidian 把所有符号当纯文字渲染，格式全崩。

**错误原因**：错误地传了 JSON `{"content": "..."}` 而非纯文本。

**正确做法**：PUT 端点直接接收纯文本内容，不需要 JSON 包装。
- ❌ 错误：`PUT body = {"content": "# 标题\n正文"}`
- ✅ 正确：`PUT body = "# 标题\n正文"`（纯字符串）

**预防措施**：优先使用文件系统直接读写（Write/Edit 工具），比调 REST API 更可靠，除非必须在 Obsidian 运行时实时同步。

### ⚠️ 教训 2：删除旧文件前必须先验证新文件

**问题现象**：执行"重命名"时采用了"删旧→建新"的顺序，新文件内容写坏了，旧文件也没了，差点丢数据。

**错误原因**：删旧文件的顺序在前，没有先确认新文件内容是否正确。

**正确做法**：
1. 创建新文件
2. 读取新文件，验证内容正确
3. 确认无误后再删除旧文件
4. 如果新文件有问题，保留旧文件，重新操作

**预防措施**：永远先建后删，不能先删后建。

### ⚠️ 教训 3：简单重命名不要绕弯子

**问题现象**：改文件名前面的一个数字，却走了"读内容→调 API 创建新文件→删旧文件"的复杂流程，层层出错。

**错误原因**：选择了复杂路径，而不是直接重命名文件系统。

**正确做法**：直接文件系统重命名，`os.rename(old_path, new_path)` 一步到位。
- 只有 Obsidian 正在运行且文件被锁定时才需要用 API，否则直接改文件系统更稳。

**预防措施**：遇到重命名需求，第一反应是文件系统操作，而不是 API 操作。

### ⚠️ 教训 4：文件内容"堆在一起"的典型原因

**问题现象**：Obsidian 笔记出现以下症状：所有文字挤在一行，`\n` 显示为文字而非换行，表格/列表/标题全失效。

**错误原因**（常见 3 种）：
1. 写了 JSON 字符串到文件（`{"content":"..."}`）
2. Python 字符串被二次转义（`\\n` 写成 `\n`）
3. 编码错误（GBK 写 UTF-8 文件导致中文乱码）

**正确做法/修复方法**：用 Python 的 `json.loads()` 解析文件内容，提取 `"content"` 字段，重新写回。

**预防措施**：写文件前，先在本地预览内容格式是否正确，确认无误再写入。
